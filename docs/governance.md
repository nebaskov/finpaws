# Governance

## Risk Register

| Риск | Вероятность | Влияние | Детект | Защита | Остаточный риск |
|--|----|---|--|--|-----|
| Галлюцинации LLM: неверная категоризация или расчёт | Средняя | Высокое | Eval-тесты на тестовом наборе транзакций; мониторинг tool-call accuracy | Арифметика вынесена в детерминированные tools; категоризация с подтверждением пользователя при низкой уверенности | Единичные ошибки в edge-кейсах |
| Prompt injection через текст транзакции | Низкая | Высокое | Логирование всех промптов; анализ аномальных tool-calls | Валидация входных данных; system prompt с явными границами; ограничение доступных tools | Остаётся при сложных атаках |
| Недоступность LLM API | Средняя | Высокое | Health-check, таймауты | Retry с exponential backoff; graceful degradation (показать сырые данные без рекомендаций) | Полная недоступность при длительном outage |
| Недоступность API курсов валют | Низкая | Среднее | Таймаут + HTTP status мониторинг | Кэширование последнего известного курса; предупреждение пользователя о неактуальности | Неточная конвертация при устаревшем курсе |
| Утечка финансовых данных пользователя через LLM API | Низкая | Высокое | Аудит отправляемых промптов | Минимизация данных в промптах (агрегаты вместо сырых записей); локальное хранение | Данные проходят через LLM-провайдера |
| Некорректный финансовый совет | Средняя | Среднее | Eval-набор с эталонными рекомендациями | Дисклеймер: "не является финансовой консультацией"; ограничение scope рекомендаций | Пользователь может принять совет буквально |
| Утечка/подбор JWT-токена | Низкая | Высокое | Мониторинг аномальных сессий; логирование auth-событий | Короткий TTL токенов; хэширование паролей (bcrypt); HTTPS | Компрометация при утечке секрета подписи |
| Доступ к чужим данным (IDOR) | Низкая | Высокое | Тесты на изоляцию данных между пользователями | Фильтрация по user_id на уровне каждого запроса к БД; row-level security в PostgreSQL | Ошибка в новом endpoint без фильтра |

## Политика логов

- Все обращения к LLM логируются: timestamp, тип запроса (без полного текста промпта в production).
- Логи tool-calls: название инструмента, параметры, результат, время выполнения.
- Финансовые данные пользователя хранятся в PostgreSQL с изоляцией по user_id. В логи попадают только агрегированные метрики (сумма по категории), не отдельные транзакции.
- Логи не содержат персональных данных (имя, контакты). При PoC персональные данные не собираются.
- Ротация логов: хранение 30 дней, затем удаление.

## Персональные данные

Система не передаёт данные третьим сторонам, кроме LLM-провайдера. Финансовые данные передаются в LLM в минимальном объёме, необходимом для генерации ответа.

- Пароли хранятся в виде bcrypt-хэшей, не в открытом виде.
- Данные пользователей изолированы на уровне строк в PostgreSQL (row-level security).
- Возможность полного удаления аккаунта и всех данных пользователя.
- При развитии за пределы PoC потребуется шифрование БД at rest и явное согласие на обработку данных.

## Защита от injection

- System prompt содержит явные инструкции о границах поведения агента.
- Пользовательский ввод не встраивается в system prompt напрямую - передаётся как user message.
- Набор доступных tools ограничен и фиксирован: агент не может выполнять произвольный код.
- Валидация параметров tool-calls: суммы - числа, валюты - из фиксированного списка, категории - из справочника.

## Подтверждение действий

- Удаление транзакций и сброс бюджета требуют явного подтверждения пользователя.
- Изменение целей накопления - агент переспрашивает перед применением.
- Массовые операции (импорт, пересчёт) выполняются только по явному запросу.
