# Product Proposal

## Обоснование идеи

Личное финансовое планирование - задача, которую большинство людей откладывает или выполняет нерегулярно. Основные причины: рутинность ручного ввода, сложность анализа, отсутствие обратной связи. Табличные приложения и трекеры расходов решают задачу учёта, но не помогают с принятием решений.

FinPaws решает эту проблему, предоставляя агента, который:
- ведёт диалог на естественном языке для ввода и анализа транзакций,
- запоминает финансовый контекст пользователя между сессиями,
- самостоятельно обращается к внешним API и инструментам для обогащения данных,
- формирует рекомендации на основе истории трат и заданных целей.

Кошачья стилистика снижает психологический барьер при работе с финансами и повышает вовлечённость.

## Цель и метрики

Цель: создать PoC агентной системы, способной вести персональный бюджет через диалог с сохранением контекста.

### Продуктовые метрики
- Корректность категоризации транзакций: >= 90% точность
- Полнота бюджетного плана: план покрывает все заданные категории расходов
- Удовлетворённость рекомендациями: субъективная оценка релевантности на тестовых сценариях

### Агентские метрики
- Tool-call accuracy: доля корректных вызовов инструментов от общего числа
- Task completion rate: доля успешно завершённых пользовательских запросов
- Среднее количество шагов агента на задачу

### Технические метрики
- p95 latency ответа агента: < 10 сек
- Успешность обращений к внешним API: >= 95%

## Сценарии использования

### Базовые
1. Пользователь сообщает о расходах в свободной форме ("потратил 1500 на продукты"), агент категоризирует и сохраняет.
2. Пользователь запрашивает сводку за неделю/месяц - агент формирует отчёт с разбивкой по категориям.
3. Пользователь ставит цель накопления ("хочу накопить 100к за полгода") - агент рассчитывает план и отслеживает прогресс.
4. Агент предупреждает о приближении к лимитам бюджета по категориям.
5. Пользователь загружает CSV/Excel с транзакциями за период - агент парсит, категоризирует и добавляет в базу.

### Edge-кейсы
- Транзакция в иностранной валюте: агент запрашивает курс через API и конвертирует.
- Неоднозначная категория ("кофе с коллегами" - еда или развлечения): агент уточняет у пользователя и запоминает решение.
- Противоречивые данные: пользователь указывает доход меньше суммы расходов - агент сигнализирует о расхождении.
- Недоступность внешнего API курсов: агент использует кэшированный курс и предупреждает о неактуальности.
- Загрузка файла с нестандартными колонками или форматами дат: агент пытается определить маппинг автоматически, при неудаче просит пользователя уточнить.
- Дубликаты при повторном импорте: агент детектирует совпадения по дате+сумме+описанию и предлагает пропустить.

## Ограничения

### Технические
- SLO: доступность агента 95% (PoC, локальный запуск)
- p95 latency: < 10 сек на ответ (зависит от LLM-провайдера)
- Хранилище: PostgreSQL, данные пользователей изолированы на уровне строк

### Операционные
- Бюджет на LLM API: ограничен тестовыми запросами (оценка ~$5-10 на этап разработки)
- Внешние API: бесплатные тарифы (курсы валют)
- Команда: 1 человек

## Архитектурный набросок

```
User (CLI/API) -> Auth (JWT) -> Orchestrator Agent
                  |
                  +-> LLM
                  |
                  +-> Tools:
                  |      +- transaction_manager (CRUD транзакций)
                  |      +- budget_planner (лимиты, планы, цели)
                  |      +- analytics (сводки, тренды, прогнозы)
                  |      +- currency_converter (внешний API курсов)
                  |      +- file_importer (парсинг CSV/Excel)
                  |
                  +-> Auth:
                  |      +- registration / login (JWT)
                  |
                  +-> Memory:
                  |      +- PostgreSQL (пользователи, транзакции, категории, цели)
                  |      +- Conversation history (контекст диалога)
                  |      +- User preferences (категории, стиль)
                  |
                  +-> Knowledge base (гибрид):
                  |      +- OpenSearch: финансовые советы, справочники (векторный поиск)
                  |      +- MinIO (S3): шаблоны бюджетов, исходные документы базы знаний
                  |      +- PostgreSQL: пользовательские правила категоризации, предпочтения
                  |
                  +-> Storage:
                         +- MinIO (S3): загруженные файлы (CSV/Excel), экспорты отчётов
```

## Data flow

### Делегируется LLM/Agent
- Понимание намерения пользователя (intent recognition)
- Извлечение суммы, категории, валюты из свободного текста
- Генерация рекомендаций и советов
- Выбор инструмента и формирование параметров вызова
- Формулировка ответов

### Не делегируется LLM
- Арифметика бюджета (суммы, остатки, проценты) - детерминированные вычисления в tools
- CRUD-операции с базой данных
- Запросы к внешним API (курсы валют)
- Валидация данных (формат суммы, допустимость валюты)
- Загрузка/скачивание файлов из MinIO
